# /sdd-run — 스펙 주도 개발 워크플로

한 기능을 **스펙 → 테스트 → 코드** 순서로, 각 단계를 사용자 컨펌과 함께 완성한다.
이 스킬은 `CLAUDE.md`의 "스펙 · 테스트 · 코드는 항상 Sync" 원칙을 절차로 강제한다.

핵심 규칙:
- **단계를 건너뛰지 않는다.** 각 게이트를 통과해야 다음으로 넘어간다.
- **컨펌이 필요한 게이트에서는 반드시 멈추고 사용자 승인을 받는다.**
- 한 번에 **하나의 기능**만 다룬다. 범위가 커지면 쪼개서 제안한다.

---

## 1단계 — 요구사항 명확화 (핑퐁)

사용자와 주고받으며 요구사항을 구체화한다.

- 무엇을, 왜, 어떤 입력/출력/엣지케이스를 원하는지 질문한다.
- 모호하거나 가정이 필요한 지점은 **추측하지 말고 물어본다.**
- 기존 스펙(`docs/spec/`)·코드와 충돌하거나 영향을 주는 부분을 짚는다.
- 정리되면 요구사항을 한두 문단으로 요약해 **"이대로 진행할까요?"** 확인받는다.

➡️ 요구사항 합의 전에는 2단계로 넘어가지 않는다.

## 2단계 — 스펙 수정 (컨펌 게이트)

합의된 요구사항에 맞춰 스펙 문서를 **먼저** 수정한다.

- 대상: `docs/spec/`의 `entities/` · `features/` · `data/` 중 해당 문서. 새 문서면 생성 후 `SPEC.md` 목차에 링크 추가.
- **실제로 구현할 내용만** 적는다. 미구현/유예 항목은 `TODO` / `미구현` / `(미사용)`으로 표시.
- **테스트 시나리오 섹션을 스펙에 추가**한다. 형식 예:
  ```
  ## 테스트 시나리오
  - [정상] 입력 X → 결과 Y
  - [경계] 입력이 최소/최대일 때 → ...
  - [예외] 잘못된 입력 → ...
  ```
  이 시나리오가 3단계 Red 테스트의 근거가 된다.
- 스펙 변경안(diff 또는 요약)을 제시하고 **"스펙 이대로 확정할까요?"** 컨펌받는다.

➡️ **스펙 컨펌이 완료되어야** 3단계로 넘어간다. 미확정 스펙으로 테스트/코드를 쓰지 않는다.

## 3단계 — Red 테스트 작성

확정된 스펙의 테스트 시나리오를 **실패하는 테스트**로 옮긴다.

- 시나리오 각 항목을 테스트 케이스로 작성한다.
- 아직 구현이 없으므로 테스트는 **실패(Red)해야 정상**이다. 실행해서 실제로 실패하는지 확인한다.
- ⚠️ 테스트 프레임워크가 아직 없다면(현재 프로젝트 상태) **먼저 테스트 하네스를 세팅**한다.
  Godot은 [GUT](https://github.com/bitwes/Gut) 등을 쓴다. 도입 방식은 사용자와 합의한 뒤 진행한다.
- Red 확인 결과(무엇이 왜 실패하는지)를 보고한다.

➡️ Red가 확인되면 **바로** 4단계로 넘어간다.

## 4단계 — Green 구현

테스트를 통과시키는 **최소 구현**을 한다.

- 3단계 테스트가 모두 통과(Green)할 때까지 구현한다.
- 스펙 범위를 벗어나는 기능은 넣지 않는다.
- 테스트를 다시 돌려 전부 통과하는지 확인하고 결과를 보고한다.

## 5단계 — 리팩터링

테스트가 Green인 상태를 유지하며 코드를 정리한다.

- 중복 제거, 명확한 이름, 주변 코드 스타일(주석 밀도·관용구)과 일치.
- 리팩터링 후 **테스트를 다시 돌려 여전히 Green인지 확인**한다.
- 스펙·테스트·코드가 최종적으로 일치(Sync)하는지 점검하고 마무리 요약을 보고한다.

➡️ **마지막 단계까지 끝나면 `/sdd-done`으로 마무리한다** (코드리뷰 → 테스트 → 커밋).

## 완료 체크리스트

- [ ] 요구사항 합의 완료
- [ ] 스펙 수정 + 테스트 시나리오 추가 + 컨펌 완료
- [ ] Red 테스트 작성 및 실패 확인
- [ ] Green 구현 및 전체 통과 확인
- [ ] 리팩터링 후 테스트 Green 유지
- [ ] 스펙 · 테스트 · 코드 Sync 확인
