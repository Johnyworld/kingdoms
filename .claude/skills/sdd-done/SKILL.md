# /sdd-done — SDD 마무리 워크플로

구현이 끝난 변경을 리뷰·검증·커밋으로 마무리한다. 보통 `/sdd-run` 다음에 이어서 쓴다.

핵심 규칙:
- 단계 순서를 지킨다: **Sync 점검 → 리뷰 → 테스트 → 커밋**.
- 테스트가 실패하거나 리뷰에서 치명적 이슈가 남아 있으면 **커밋하지 않는다.**

---

## 0단계 — 스펙 · 테스트 · 코드 Sync 점검

가장 먼저 세 가지가 서로 어긋난 부분이 없는지 확인한다.

- **스펙 ↔ 코드**: 스펙에 적힌 동작/속성/값이 실제 코드와 일치하는가? 코드에만 있고 스펙에 없는 기능, 스펙에만 있고 구현 안 된 항목(`TODO` 표시 없이)이 있는가?
- **스펙 ↔ 테스트**: 스펙의 테스트 시나리오가 모두 테스트로 존재하는가?
- **테스트 ↔ 코드**: 테스트가 현재 코드의 실제 동작을 검증하는가?
- 어긋난 부분이 있으면 **어느 쪽이 옳은지 판단**해서 나머지를 맞춘다. 애매하면 사용자에게 확인한다.

➡️ 셋이 일치하는 상태를 확인한 뒤 1단계로 넘어간다.

## 1단계 — 코드리뷰 3사이클 (전문 에이전트 위임)

리뷰는 직접 하지 말고 **코드리뷰 전문 에이전트에게 위임**한다. (Agent 툴의 리뷰 에이전트 또는 `/code-review` 스킬)

한 사이클 =
1. 현재 변경분(diff)을 리뷰 에이전트에 넘겨 리뷰를 받는다.
2. 지적 사항을 검토한다. 타당한 것은 수정하고, 반영하지 않은 것은 이유를 남긴다.
3. 수정했으면 관련 스펙·테스트도 함께 Sync한다. (`CLAUDE.md` 원칙)

이 사이클을 **최대 3회** 반복한다. 종료 조건:
- 3사이클을 모두 돌았거나,
- 리뷰에서 더 이상 유효한 지적이 나오지 않으면(clean) 조기 종료.

각 사이클마다 무엇이 지적됐고 어떻게 처리했는지 요약해 보고한다.

➡️ 리뷰가 마무리되면 2단계로 넘어간다.

## 2단계 — 테스트 실행

**전체 테스트**를 실행한다 (해당 기능만이 아니라 회귀 확인 목적).

- 전부 통과(Green)해야 3단계로 넘어간다.
- 실패가 있으면 원인을 고치고(필요 시 1단계 방식으로 스펙/테스트/코드 Sync 유지) 다시 실행한다.
- 실행 결과를 그대로 보고한다. 통과/실패를 얼버무리지 않는다.

## 3단계 — 커밋

테스트가 Green이고 리뷰가 정리된 상태에서만 커밋한다.

- 변경 파일을 확인하고(`git status`, `git diff`) 관련 변경만 스테이징한다.
- 커밋 메시지는 `CLAUDE.md`의 커밋 규칙을 따른다. 이번 작업에는 스펙·테스트·코드가 함께 포함됨을 반영한다.
- 푸시는 사용자가 요청할 때만 한다.

## 완료 체크리스트

- [ ] 전체 테스트 실행 및 전부 통과
- [ ] 스펙 · 테스트 · 코드 Sync 확인
- [ ] 커밋 완료 (Co-Authored-By 포함)
